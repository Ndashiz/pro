Architecture globale

LazyPO est une Single Page Application sans framework JavaScript. La navigation est gérée par un routeur maison (sidebar.js) qui swape le contenu de la page sans rechargement complet, préservant l'état des modules persistants (Focus FM, session, realtime).

Architecture de sécurité

LazyPO est hébergé sur GitHub Pages (statique, pas de serveur applicatif). Avant Mai 2026, l'authentification était 100% côté client — le HTML protégé était téléchargeable par n'importe qui via curl, le contrôle d'accès reposait uniquement sur du JavaScript qui s'exécutait après livraison.

Depuis PR #112, un Cloudflare Worker intercepte chaque requête à ndashiz.be/lazypo/* et valide un cookie de session avant que le HTML soit livré. Le contenu protégé n'est jamais servi à un visiteur non authentifié.

Authentification — auth.js

Stack auth :

  • Supabase Auth avec email/password — émet des JWT signés en ES256 (asymmetric, JWKS public).
  • Approbation manuelle via profiles.is_approved — un user signupé n'a pas accès tant qu'un admin n'a pas approuvé.
  • Cookie lazypo_jwt miroir du access_token, Path=/lazypo, SameSite=Lax, Secure (https uniquement). Lu par le Worker pour gater le HTML.

Approbation utilisateur

Un user fraîchement signupé n'a pas accès tant que profiles.is_approved = false. Le check est fait dans login.html immédiatement après signInWithPassword :

  • Si not approved → signOut immédiat + message "Pending approval"
  • Si approved → cookie posé, redirect vers return_to ou index.html

Gestion de l'avatar

Les avatars sont uploadés dans le bucket Supabase Storage avatars avec le chemin {user_id}/avatar.{ext}. Le bucket est en lecture publique, écriture authentifiée (RLS).

Caches

Deux couches de cache pour minimiser la charge :

  • JWKS cache (1h) — la liste des clés publiques de Supabase est récupérée une fois par heure, stockée dans caches.default au PoP. Un kid manquant déclenche un refresh immédiat (gère la rotation).
  • Token verdict cache (60s) — pour chaque JWT vu, on cache le verdict (200 ok ou 401 bad) pendant 60s. Un user qui charge 20 ressources avec le même JWT = 1 vérification crypto + 19 cache hits.

Coût

Cloudflare Workers free tier : 100 000 requêtes/jour. LazyPO sit à quelques milliers/jour → gratuit indéfiniment.

Threat model

Liste des vecteurs d'attaque identifiés lors de l'incident de Mai 2026, et comment chacun est désormais fermé.

# Vecteur Avant Mitigation
V1 auth-gate.js révélait le contenu après 3s peu importe l'état d'auth 3s de patience → contenu visible même sans session valide Worker bloque la livraison HTML. auth-gate.js redirige vers login après 5s au lieu de révéler.
V2 auth-gate.js acceptait n'importe quel localStorage avec access_token Forge en JS : localStorage.setItem(..., '{"access_token":"x"}') Vérification stricte côté client (3-part JWT + exp futur). Surtout : le Worker valide la signature ES256 côté serveur — un faux cookie ne passe pas.
V3 _showModuleLockOverlay = simple <div> z-indexé DevTools → supprimer le div → accès au contenu Partiellement mitigé : le Worker garantit que tu es au moins authentifié. Module-level gate côté client reste DevTools-bypassable. Mitigation principale = RLS Supabase sur la DATA.
V4 requireModule() async dans DOMContentLoaded → DOM déjà parsé Contenu protégé visible 100-500ms avant la décision auth Le HTML n'est pas livré du tout si pas authentifié (Worker). Le problème de race disparaît.
V5 _fetchAllowedModules fail-open à ['quiz'] sur erreur Supabase Supabase down → tous les users obtenaient l'accès au module quiz Note interne : à durcir vers [] en cas d'erreur. Tracking issue séparée.
V6 DEV_SESSION activé automatiquement sur tout hostname localhost-class Pas exploitable directement en prod, mais le code restait actif et pouvait être élargi par erreur Bypass dev nécessite désormais opt-in explicite : window.__ENABLE_LOCAL_BYPASS=true OU sessionStorage 'lazypo:enableLocalBypass'. Plus jamais activé automatiquement.

Risques résiduels (assumés)

  • Cookie non HttpOnly — Supabase JS le pose côté client, donc impossible de mettre HttpOnly. Une XSS pourrait exfiltrer le JWT. Mitigation : CSP stricte (à formaliser), pas d'innerHTML avec contenu user.
  • Latence de révocation cross-device — sign out sur appareil A, le JWT reste valide sur appareil B jusqu'à exp (~1h). Acceptable pour notre modèle de menace.
  • Module gate client-only — un user authentifié peut DevTools-supprimer le lock overlay et voir une page d'un module auquel il n'a pas accès. La donnée reste protégée par RLS Supabase ; seules les coques HTML sont visibles.
  • Sécurité de Cloudflare elle-même — toute compromission du compte CF ou de la zone ndashiz.be permettrait de retirer le Worker. Mitigation : 2FA obligatoire sur le compte CF.

Algorithme SM-2 — Répétition espacée

L'algorithme SuperMemo 2 calcule l'intervalle optimal entre deux révisions d'un mot, selon la qualité de la réponse de l'utilisateur.

Tolérance aux fautes de frappe

Les réponses avec 1 caractère de différence (distance de Levenshtein ≤ 1) sont acceptées comme correctes, mais avec un score de qualité légèrement réduit. Cela évite les faux négatifs sur des erreurs de saisie mineures.

Mai 2026 · PR #112 — 🔒 Server-side auth gate (Cloudflare Worker)

  • Incident : login bypass récurrent (3ème fois). Root cause = site statique + auth 100% côté client = HTML protégé téléchargeable par tout visiteur.

  • Diagrammes : voir section ★ Architecture de sécurité, Cloudflare Worker et Threat model.
  • Coût : $0 — free tier Cloudflare Workers (100k req/jour, on est très en dessous).
